# Interview question bank — by system type

Ask ONE at a time. Follow the answer, not the list. Add questions that worked
after every run.

## Universal (always usable)

1. Walk me through the last time you did this, start to finish — the actual
   last time, not the ideal version.
2. What's the first correction you'd give an assistant who tried this — the
   thing you'd never think to say up front?
3. Which step do you dread? Which step do you always forget?
4. How do you know when the output is done vs. needs another pass?
5. What would break if you skipped step N? (Ask for the step they described
   most vaguely.)
6. What part exists only in your head — no file, no doc, no template?
7. When this goes badly, what does badly look like? Show me a real bad output
   if one exists.
8. Show me the best output this system ever produced. What made it the best?

## Content / writing systems

- Where do ideas enter — and where do good ones die before publishing?
- What's your actual quality bar: what have you refused to publish, and why?
- Whose voice is this in? Show me a paragraph that nails it and one that
  misses it.

## Coding / vibe-coding systems

- What do you check before you trust generated code — run it, read it, test?
- What class of bug bites you repeatedly?
- When do you restart a session vs. keep iterating in place?

## Sales / outreach / follow-up systems

- What triggers a follow-up — time elapsed, an event, a feeling?
- What's the line you never cross (tone, frequency, discount)?
- Which message template actually converts, and how do you know?

## Media / editing systems

- What's the fixed skeleton every output shares vs. what varies per piece?
- What's the export/delivery checklist you've memorized but never written?

## Engineering domains (ML research, agentic, DevOps, distributed, systems)

Domain-specific questions live with their pipelines in
`domain-playbooks/<domain>.md` — each playbook's "Known failure modes"
section doubles as an interview source: ask "when did [failure mode N] last
happen to you?" for each one.

## Closing question (always ask last)

- If a cheap model ran this tomorrow without you, which single step do you
  trust it with LEAST? (That answer is usually the real bottleneck.)
